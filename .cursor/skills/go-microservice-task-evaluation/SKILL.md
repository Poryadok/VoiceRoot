# Оценка выполнения задачи (Go-микросервис)

Скилл для **финальной проверки** изменений в Go-сервисе: не заменяет разработку и TDD, но задаёт полный контур «готово / не готово» и приоритеты доработок.

## Когда применять

- Пользователь просит **оценить выполнение**, **проверить готовность**, **аудит** задачи по конкретному сервису или модулю `src/backend/<service>/`.
- Перед merge, после реализации, или для пост-ревью без правок кода (только вердикт и список действий).

## Входные данные (зафиксировать в начале)

1. **Путь к сервису** (например `src/backend/messaging/`) и затронутые пакеты (`pkg`, `protos`, другой сервис).
2. **Формулировка задачи** и ссылки на требования: issue/PR, фрагмент `docs/features/`, `docs/microservices/<service>.md`, ADR — что есть.
3. **Граница сервиса**: что считается «внутри» задачи (HTTP/gRPC, БД, очереди, клиенты к другим сервисам).

Если граница неясна — явно перечислить предположение в отчёте.

---

## Порядок работы (сначала факты, потом суждения)

1. **Собрать контекст из репозитория** (релевантные `docs/` для этого сервиса: `MICROSERVICES.md`, карточка в `docs/microservices/`, при смене данных — `DATA_STORES.md`, `DATA_MODEL.md`, сквозные требования — `ARCHITECTURE_REQUIREMENTS.md`).
2. **Запустить проверки** (см. раздел «Команды») и зафиксировать вывод (успех/ошибка, ключевые строки).
3. **Пройти чеклист** ниже по каждому блоку; для каждого пункта: выполнено / частично / нет + краткое обоснование.
4. **Сформировать отчёт** по шаблону в конце файла.

Не выдумывать продуктовое поведение: если требование не в доках и не в задаче — отметить как **пробел**, не как «ошибка реализации».

---

## Команды (минимум для Go-сервиса)

Выполнять из каталога сервиса (`src/backend/<service>/`), если не указано иначе. В PowerShell цепочки разделять **`;`**, не `&&`.

| Цель | Команда |
|------|---------|
| Юнит-тесты | `go test ./...` |
| Race (если уместно и доступен CGO для сервиса; для Gateway в этом репо — см. `docs/TESTING.md`) | `CGO_ENABLED=1 go test -race ./...` |
| Покрытие (сводка) | `go test ./... -coverprofile=coverage.out ; go tool cover -func=coverage.out` |
| Сборка бинаря / проверка компиляции | `go build -o NUL .` (Windows) или `go build ./...` |
| Линтер | `golangci-lint run` из корня монорепо или с конфигом репозитория |
| Монорепо Voice (паритет с CI) | из корня: `make build-all` (в т.ч. golangci-lint по модулям и `go test -race` для gateway) или точечно `go test` в `src/backend/pkg` и целевом сервисе — см. `Makefile`, `.github/workflows/ci.yml` |
| Protobuf (если менялись `protos/`) | `buf lint`, `buf format -d --exit-code`, на PR-ветке — `buf breaking` к базе |

Если команда недоступна (нет Docker, нет `golangci-lint`) — указать в отчёте и заменить доступной эквивалентной проверкой.

## 1. Работоспособность и соответствие задаче

- [ ] Сервис **компилируется**; нет регрессий в зависимостях (`go mod tidy` при необходимости без лишних изменений).
- [ ] Поведение **совпадает с задачей** и с контрактом в `docs/` (REST/gRPC схемы, коды ошибок, идемпотентность где задано).
- [ ] **Конфигурация**: новые env-переменные описаны в `.env.example` / деплой-доках при необходимости; дефолты безопасны для prod.
- [ ] **Миграции БД** (если есть): применимы, обратимость/риски отмечены; соответствие `DATA_STORES.md` / владельцу данных.
- [ ] **Health / readiness** (если приняты в сервисе): эндпоинты не лгут; зависимости отражены корректно.

---

## 2. Тесты и покрытие

- [ ] **`go test ./...` зелёный** для затронутых модулей.
- [ ] Тесты проверяют **поведение из спецификации**, а не только «код вызвался»; табличные тесты там, где много вариантов (`docs/TESTING.md`: `testing` + `testify`).
- [ ] **Граничные случаи**: пустые входы, ошибки хранилища/сети (моки), таймауты, отмена контекста — по релевантности задаче.
- [ ] **Интеграционные** тесты: если задача касается БД/Redis/gRPC — есть ли реальный слой или осознанный gap с планом добавления (как в `docs/TESTING.md`).
- [ ] **Покрытие**: процент не цель сам по себе; важно, что **критические ветки** и новая логика покрыты. Указать слабые файлы/функции по `go tool cover -func` при споре.
- [ ] **Race**: для конкурентного кода — прогон с `-race` где принято в репозитории.
- [ ] Тесты **детерминированы** (без флака от времени/порядка без явной нужды).

---

## 3. Качество кода (читаемость, ошибки, поддержка)

- [ ] **Обработка ошибок**: не глотаются; контекст (`fmt.Errorf` с `%w`) где нужно; корректные коды на границе API.
- [ ] **Контексты**: `context.Context` протянут в IO и отменяемые операции.
- [ ] **Имена и размер функций**: понятные границы ответственности; без копипасты с минимальным отличием.
- [ ] **Логирование**: структурировано/уровни; нет секретов в логах; шум не забивает сигнал.
- [ ] **Зависимости**: новые библиотеки оправданы; лицензии и размер не вредят продукту.
- [ ] Соответствие **локальным правилам** репозитория (`golangci-lint`, соглашения по пакетам).

---

## 4. Технологии и операционка

- [ ] Версия **Go** и образов согласованы с репозиторием (`Makefile`, Dockerfile сервиса).
- [ ] **Dockerfile** сервиса: многостадийная сборка, непривилегированный пользователь где принято, минимальный runtime.
- [ ] **Метрики и трейсинг** (если в сервисе уже есть стандарт): новые пути учтены.
- [ ] **Graceful shutdown**: сигналы, drain горутин/серверов при наличии в кодовой базе.

---

## 5. Безопасность и устойчивость

- [ ] Валидация **входных данных** на границе (размеры, лимиты, санитизация идентификаторов).
- [ ] **Авторизация**: не обходятся проверки; права соответствуют модели (роли/ACL — по доке сервиса).
- [ ] **Секреты** не в коде; используются env/vault-паттерн репозитория.
- [ ] **Rate limiting / abuse** — на Gateway или в сервисе согласно архитектуре; не дублировать без причины.
- [ ] Зависимости от внешних сервисов: **таймауты**, ретраи с backoff где уместно, идемпотентные клиенты.

---

## 6. Архитектура и границы микросервиса

- [ ] Логика **лежит в правильном сервисе** (`MICROSERVICES.md`, `PLAN.md` — соответствие модулей); нет тихого переноса ответственности другому сервису без согласования.
- [ ] **Синхронные вызовы** vs **события**: не нарушен выбранный стиль; нет лишних синхронных цепочек, которые должны быть асинхронными по доке.
- [ ] **Realtime vs Messaging** (для этого продукта): WS-последовательность и догрузка сообщений — не перепутаны (`ARCHITECTURE_REQUIREMENTS.md`).
- [ ] **Идентификаторы и ссылки между сервисами** — по `DATA_MODEL.md` (типы ID, внешние ключи логически, не физический FK между сервисами без оснований).
- [ ] **gRPC/REST контракты**: изменения совместимы; protobuf не ломает потребителей без версионирования/согласования.
- [ ] **Gateway**: новые маршруты/прокси согласованы с `docs/microservices/api-gateway.md` при изменении публичного API.

---

## 7. Документация и сопровождение

- [ ] Публичное поведение отражено в **`docs/`** там, где договор репозитория требует (контракт сервиса, фича, миграции).
- [ ] **CHANGELOG / релизные заметки** — по процессу команды (если принято).
- [ ] Сложные места: короткий комментарий «почему так», а не эссе.

## Шаблон отчёта

```markdown
## Оценка задачи: <сервис> — <кратко задача>

**Вердикт:** Готово / Готово с замечаниями / Не готово

### Команды
- `go test ./...`: OK/FAIL (выдержка)
- `golangci-lint`: OK/FAIL
- (прочее)

### Сильные стороны
- ...

### Замечания по приоритету
1. **[Критично]** ...
2. **[Важно]** ...
3. **[Желательно]** ...

### Пробелы в требованиях
- ...

### Рекомендованные следующие шаги
- ...
```
