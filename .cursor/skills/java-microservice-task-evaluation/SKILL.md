# Оценка выполнения задачи (Java-микросервис)

Скилл для **финальной проверки** изменений в Java-сервисе (в этом репозитории — **Auth**, Spring Boot): задаёт контур «готово / не готово» и приоритеты доработок. Не заменяет TDD и разработку.

## Когда применять

- Оценка выполнения, готовность к PR, аудит по модулю вроде `src/backend/auth/`.
- Любой другой Java-микросервис в монорепо — те же принципы; команды подстроить под **Maven** или **Gradle** проекта.

## Входные данные (зафиксировать в начале)

1. **Путь к модулю** (например `src/backend/auth/`) и затронутые артефакты (`protos`, общие контракты с Go).
2. **Формулировка задачи** и ссылки: issue/PR, `docs/microservices/auth-service.md`, фичи, ADR.
3. **Граница сервиса**: REST, gRPC, БД, Redis, интеграции с Gateway/другими сервисами.

Если граница неясна — явно указать предположение в отчёте.

---

## Порядок работы

1. **Контекст из `docs/`**: `MICROSERVICES.md`, карточка сервиса в `docs/microservices/`, при данных — `DATA_STORES.md`, `DATA_MODEL.md`, сквозные правила — `ARCHITECTURE_REQUIREMENTS.md`.
2. **Запустить проверки** (раздел «Команды»); зафиксировать OK/FAIL и выдержки лога.
3. **Чеклисты** по блокам: выполнено / частично / нет + короткое обоснование.
4. **Отчёт** по шаблону в конце.

Не придумывать продуктовое поведение вне доков и задачи — такие пункты → **пробел / needs decision**.

## Команды

Рабочий каталог — **корень Java-модуля** (например `src/backend/auth/`). В PowerShell цепочки: **`;`**, не `&&`.

### Maven (как в Voice Auth)

| Цель | Команда |
|------|---------|
| Тесты | `.\mvnw.cmd test` (Windows) или `./mvnw test` (Unix) |
| CI-стиль (batch) | `mvn -B test` (как в `make auth-test-ci`) |
| Сборка без тестов (только компиляция) | `mvn -B package -DskipTests` — только если нужно быстро проверить компиляцию; для «готово» нужны тесты |
| Полный verify | `mvn -B verify` — если в `pom.xml` подключены доп. фазы (интеграционные тесты, плагины) |
| Юнит vs интеграция (Maven) | по соглашению в `pom`: обычно Surefire — `*Test.java`, Failsafe — `*IT.java` (или суффикс из конфига); для «готово» не пропускать фазу Failsafe, если IT есть |
| Один модуль из реактора | `mvn -pl <module> -am test` |
| Один класс / метод | `mvn -B -Dtest=ClassName#method test` — для локальной диагностики |
| Покрытие (JaCoCo и т.п.) | только если настроено в `pom.xml`: например `mvn -B test jacoco:report` — иначе отметить в отчёте «плагин не заведён» |

### Gradle (если проект на Gradle)

| Цель | Команда |
|------|---------|
| Тесты | `.\gradlew.bat test` или `./gradlew test` |
| Интеграционные (Gradle) | `integrationTest` / `verify` — имя задачи по `build.gradle` |
| Проверки качества | `check` или `verify` — по `build.gradle` |

### Статический анализ

- Запускать **то, что подключено в репозитории** (Checkstyle, SpotBugs, PMD, Spotless, Error Prone, `mvn -DskipTests compile` с annotation processors). Если ничего нет — явно написать в отчёте и при желании предложить добавить в отдельной задаче.

### Монорепо Voice

- Из корня: **`make auth-test-ci`** — Docker + `mvn -B test` в `src/backend/auth/` (см. `Makefile`).
- Полный конвейер: **`make build-all`** (включает Auth test).
- Менялись общие **protobuf** в `protos/` — см. также скилл/процесс для **buf** (`go-microservice-task-evaluation` или `docs/TESTING.md`).

Если Docker/JDK недоступны — указать в отчёте и выполнить эквивалент на хосте (`mvnw` + локальный JDK, версия как в `pom.xml` / CI-образе).

## 1. Работоспособность и соответствие задаче

- [ ] Проект **собирается** (`mvn package` / `mvn verify` по принятому порогу); зависимости осмысленны, без случайных SNAPSHOT-ов без обоснования.
- [ ] Поведение **совпадает с задачей** и контрактом в `docs/` (эндпоинты, коды HTTP, тело ошибок, gRPC контракты).
- [ ] **Конфигурация** (`application.yml` / env): новые ключи документированы; секреты не в репозитории; профили `dev`/`prod` не смешивают небезопасные дефолты в prod.
- [ ] **Миграции** (Flyway/Liquibase): порядок, идемпотентность, откат/риски; соответствие `DATA_STORES.md` и владельцу данных.
- [ ] **Actuator / health** (если есть): readiness отражает реальные зависимости (БД, Redis).

## 2. Тесты и покрытие

У Java/Spring **больше поверхностей**, чем у Go: ниже не «всё из мира JUnit», а **расширенный минимум** для микросервиса; то, чего нет в `pom.xml`/`build.gradle`, отмечать в отчёте как gap, а не как автоматический fail.

База по репозиторию: `docs/TESTING.md` (**JUnit 5**, **Spring Boot Test**, **Testcontainers** для PostgreSQL при необходимости реальной схемы).

### Слои, контекст Spring, запуск

- [ ] **`mvn test` и при наличии IT — полный `mvn verify`** зелёные; не считать задачу проверенной, если в проекте заведён Failsafe/`integrationTest`, а прогнали только `test`.
- [ ] **Слой теста соответствует слою кода**: `@WebMvcTest` / `WebTestClient` для web, `@DataJpaTest` / `@JdbcTest` для persistence, `@JsonTest` для (де)сериализации, **узкий** `@SpringBootTest` только при необходимости полного контекста.
- [ ] **`@DirtiesContext`**: нет злоупотребления «на каждый метод» без причины (медленно, скрывает плохую изоляцию).
- [ ] Поведение проверяется **снаружи** (HTTP/JSON, статус, заголовки, тело ошибки), а не только факт вызова `@MockBean`.

### Данные, миграции, Testcontainers

- [ ] **Testcontainers**: свойства БД/Redis через `@DynamicPropertySource`, `@ServiceConnection` (Spring Boot 3.1+) или эквивалент; контейнеры **переиспользуются** между тестами/классами где принято (`@Container static`, общий suite), лишние полные старты кластера не на каждый класс без нужды.
- [ ] **H2/in-memory**: если вместо Postgres — явные ограничения (диалект, типы, функции) не ломают смысл теста; иначе Testcontainers или отдельный профиль.
- [ ] **Flyway/Liquibase** в тестах: тот же порядок миграций, что в CI; тестовые данные — `@Sql`, фикстуры, object mother / builder, а не «магические» ID без объяснения.

### Внешние вызовы, gRPC, асинхронность

- [ ] **Внешний HTTP**: стаб (**WireMock**, OkHttp **MockWebServer**) или тестовый контрактный стенд; нет обращений к прод/staging из CI.
- [ ] **gRPC**: in-process server/client или `grpc-testing` (в Auth уже есть зависимость) — проверка дедлайнов, `Status`, отмена.
- [ ] **Async / события / listeners**: ожидание через **Awaitility** или реактивный **`StepVerifier`** (WebFlux) — не `Thread.sleep` без жёсткой причины.

### Утверждения, таблицы, безопасность в тестах

- [ ] Читаемые ассерты (**AssertJ** и т.п.) на доменных структурах и JSON.
- [ ] Много вариантов одной логики — **`@ParameterizedTest`** / `@MethodSource` (или аналог в JUnit 4 legacy), без копипасты классов.
- [ ] **Spring Security в тестах**: `@WithMockUser` / `@WithUserDetails` / `SecurityMockMvcRequestPostProcessors` согласованы с реальными ролями/скоупами; нет ложнозелёных тестов из-за полностью отключенной security.
- [ ] **Граничные случаи API**: Bean Validation, 400/401/403/404/409, пустые и слишком большие тела — по релевантности задаче.

### Покрытие, контракты, архитектурные тесты

- [ ] **JaCoCo** (если подключён): пороги не обходятся; excludes для generated-кода — осмысленны.
- [ ] **Consumer-driven контракты (Pact и т.п.)**: в Voice **не требовать**, пока это не зафиксировано в `docs/` (см. `docs/TESTING.md` про Pact); если в другом репо включено — проверить публикацию артефактов и CI.
- [ ] **ArchUnit** (если заведён): новые нарушения слоёв/импортов между пакетами ловятся.

### Стабильность CI

- [ ] Нет **флака**: минимум `@Order`; нет shared **mutable static** между тестами; при **JUnit 5 parallel** — изоляция контекста/ресурсов явно продумана.
- [ ] Таймауты тестов и контейнеров разумны; нет зависимости от локального часового пояса без `Clock` injection.

## 3. Качество кода (Java / Spring)

- [ ] **Исключения**: не глотаются; доменные ошибки не утекают как 500 без необходимости; логирование с контекстом (request id / user id без PII-нарушений).
- [ ] **Транзакции**: `@Transactional` на правильном слое; границы read-only vs write; нет длинных транзакций с внешними вызовами внутри.
- [ ] **Валидация**: DTO + `jakarta.validation` на входе API.
- [ ] **Потокобезопасность**: для кэшей, пулов, async — корректная модель (Executor, `@Async`, virtual threads — по выбранному стандарту проекта).
- [ ] **Логи**: уровни, структура; пароли и токены не логируются.
- [ ] **Зависимости**: BOM Spring Boot согласован; обновления крупных библиотек (security, JWT) осознанны.

---

## 4. Технологии и операционка

- [ ] **JDK** и **Spring Boot** версии совпадают с `pom.xml` / CI (`Makefile` образ `MAVEN_IMAGE`, `.github/workflows/ci.yml`).
- [ ] **Dockerfile**: слои, non-root где принято, healthcheck при наличии стандарта.
- [ ] **Метрики / трейсинг** (Micrometer, OpenTelemetry): новые пути учтены, если это уже принято в сервисе.
- [ ] **Graceful shutdown**: корректное завершение веб-сервера и фоновых задач (Spring Boot 3.x hooks).

---

## 5. Безопасность (особенно Auth)

- [ ] **Spring Security** (или принятая модель): все новые эндпоинты в нужной цепочке; нет «перехватчика мимо» security.
- [ ] **Пароли**: хеширование (BCrypt и т.д.), соль/стоимость по стандарту проекта; не хранить в открытом виде.
- [ ] **JWT**: issuer, audience, TTL, refresh/blacklist — по `docs/microservices/auth-service.md` и `ARCHITECTURE_REQUIREMENTS.md`; ключи и алгоритмы не ослаблены «для удобства».
- [ ] **Секреты** из env/secret store; не в git.
- [ ] **CORS / CSRF** — по модели API (BFF, только server-to-server и т.д.).
- [ ] Внешние HTTP/gRPC клиенты: **таймауты**, лимиты, идемпотентность где нужно.

---

## 6. Архитектура и границы микросервиса

- [ ] В Voice целевой **Auth — Java** (`MICROSERVICES.md`); не переносить ответственность Auth на Go без явного решения в задаче/доках.
- [ ] Границы **Auth vs Gateway**: валидация JWT на Gateway; что остаётся в Auth (выдача, refresh, blacklist) — без дублирования логики без причины.
- [ ] **Идентификаторы и контракты** с другими сервисами — `DATA_MODEL.md`; изменения JWT claims согласованы с потребителями.
- [ ] **gRPC/Java stubs** из `src/main/proto`: совместимость с Go-консьюмерами при изменении контрактов; при общих protos — **buf breaking** / согласование версий.
- [ ] Нет утечки **бизнес-логики других сервисов** в Auth «потому что проще».

---

## 7. Документация и сопровождение

- [ ] Публичное поведение и конфиг отражены в **`docs/`** при изменении контракта или операционных требований.
- [ ] Сложная security-логика: краткое пояснение в коде или в доке сервиса, а не только в чате PR.
