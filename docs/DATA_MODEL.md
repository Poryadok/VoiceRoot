## 1. Идентификаторы и границы account / profile

### Тип и формат

- Первичные ключи сущностей, которые уходят в API, JWT и в другие сервисы — **UUID** (в PostgreSQL тип `UUID`). По умолчанию генерация **UUIDv4** через `gen_random_uuid()` в миграциях/БД или эквивалент в приложении.
- **Исключение:** идентификаторы строк в `messaging_db.messages` — **только UUIDv7**, выставляемые **приложением Messaging (Go)** при вставке; ULID и прочие форматы не применяем. Детали — [data/messaging-service.md](data/messaging-service.md).
- Внутренние суррогатные ключи допускаются только там, где ID наружу не отдаётся (редкие технические таблицы); по умолчанию не вводим второй «внутренний» ключ без причины.

### Аккаунт и профиль

| Понятие                    | Где живёт                                          | Назначение                                                                                         |
|----------------------------|----------------------------------------------------|----------------------------------------------------------------------------------------------------|
| **Аккаунт** (`account_id`) | **Auth Service**, БД `auth_db`, таблица `accounts` | Логин, пароль, 2FA, refresh, email/телефон, soft delete, тип regular/guest                         |
| **Профиль** (`profile_id`) | **User Service**, БД `user_db`                     | Ник, аватар, приватность, presence, мульти-профиль — [multi-profile.md](features/multi-profile.md) |

### JWT и имена в claims

В access token (и в gRPC metadata после Gateway) используются claims из [api-gateway.md](microservices/api-gateway.md):

| Claim в JWT                  | Смысл                                  | Соответствие в БД                                                |
|------------------------------|----------------------------------------|------------------------------------------------------------------|
| `user_id`                    | Идентификатор аккаунта (сквозная auth) | **`accounts.id`** = `account_id`; имя claim исторически короткое |
| `profile_id`                 | Активный профиль для этого запроса     | **`profiles.id`** в `user_db` (детали — User Service)            |
| `roles`, `subscription_tier` | Кэш для Edge                           | Source of truth — Role / Subscription; без лишних дублей в БД    |

В **новых** миграциях Auth предпочтительно колонка **`account_id`** там, где логически ссылка на `accounts.id` (в т.ч. в `refresh_tokens`). В [ARCHITECTURE_REQUIREMENTS.md](ARCHITECTURE_REQUIREMENTS.md) в примере таблицы указано имя `user_id` — при проектировании схемы трактуем его как **тот же идентификатор, что и `account_id`**, и выравниваем имя колонки к `account_id` ради однозначности.

### Что в какой БД не кладём

- В **`auth_db`** нет профилей, ников, друзей — только всё, что относится к учётной записи и сессиям.
- В **`user_db`** нет хешей паролей и refresh tokens; хранится **`account_id`** как внешняя ссылка (UUID без FK на другой кластер БД).

## 2. Ссылки между сервисами (без FK между БД)

### Правило

- У каждого микросервиса **своя** PostgreSQL БД ([DATA_STORES.md](DATA_STORES.md)). **Между разными БД внешние ключи (FK) не задаём** — СУБД не видит чужие таблицы.
- Связь «сущность A ссылается на сущность из сервиса B» выражается **колонкой UUID** (например `account_id`, `profile_id`, `chat_id`) плюс документированное правило: *валидность и жизненный цикл обеспечивает владелец B через API / события*.

### На практике

| Колонка ссылается на     | Владелец данных     | Проверка целостности                                                     |
|--------------------------|---------------------|--------------------------------------------------------------------------|
| `account_id`             | Auth                | User/Social + Auth или события; удаление аккаунта — см. продуктовые доки |
| `profile_id`             | User                | Через API и доменные правила                                             |
| `chat_id`, `space_id`, … | Chat / Space и т.д. | UUID в чужой БД; FK только внутри своей БД                               |

Дублирование справочных полей (например кэш отображаемого имени) допускается осознанно для чтения; источник истины — сервис-владелец.

### Текстовый канал (`chat_type = channel`)

**Текстовый канал** — это тип чата в **Chat Service**: строка в **`chat_db.chats`** с `type = channel` (режим ленты: посты от сущности канала, обсуждение в тредах). Канал может существовать **без спейса** (`space_id` пустой) или **внутри спейса** (`space_id` → спейс); во втором случае **Space** хранит только **место в дереве** (категория, порядок, флаги вроде system) — см. `space_db.space_text_channel_placements` в [data/target/space_db.md](data/target/space_db.md).

- В **Messaging** / **File** / **read_receipts** поле **`chat_id` при `chat_type = channel` всегда = `chat_db.chats.id`** (как и для `dm` / `group`).
- **`chat_type`** в `messaging_db.messages` различает только **поведение** ленты и тредов, а не владельца UUID: все три типа ссылаются на **`chats`**.

**Права:** канал **без** `space_id` — настройки и членство/доступ через **Chat** (и продуктовые правила видимости). Канал **с** `space_id` — дополнительно **членство в спейсе** и **Role** (оверрайды по `channel_id` = этот же `chats.id`). Messaging не вводит FK на чужие БД и опирается на решение Gateway/оркестрации.

### Голосовые комнаты в спейсе

Идентификаторы **голосовых комнат** живут в **`space_db.voice_rooms`** (отдельно от текстового потока). К **Messaging** они по умолчанию не привязаны. Оверрайды прав Role для голоса используют **`channel_id` = `voice_rooms.id`** в том же спейсе (см. [data/target/role_db.md](data/target/role_db.md)).

## 3. Общие поля, время и soft delete

### Временные метки

- **`created_at`** — `TIMESTAMPTZ`, UTC, не обнуляется.
- **`updated_at`** — `TIMESTAMPTZ`, UTC, обновляется при изменении строки (триггер или слой приложения). Для стратегии **last write wins** по доменным правилам — см. [ARCHITECTURE_REQUIREMENTS.md](ARCHITECTURE_REQUIREMENTS.md) (конкурентные операции).

Имя колонок в БД — **snake_case**.

### Soft delete

- Где в архитектуре задано soft delete (например **удаление аккаунта** — [ARCHITECTURE_REQUIREMENTS.md](ARCHITECTURE_REQUIREMENTS.md)), используем **`deleted_at TIMESTAMPTZ NULL`**: `NULL` = активная запись, непустое значение — удалена в логическом смысле.
- Для сущностей без требования «хранить после удаления» допускается жёсткое удаление или отдельная политика — фиксировать в описании конкретной таблицы.

### Прочие повторяющиеся поля

- Вводить **единый стиль** для статусов перечислением (`status` + `CHECK` или отдельный тип), а не свободные строки — в спецификации каждой таблицы перечислить допустимые значения.
- **`id`** как UUID первичный ключ — по умолчанию для всех публичных сущностей (см. п. 1).
