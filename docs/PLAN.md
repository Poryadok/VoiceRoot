# Voice — План разработки

> Каждая фаза — **один ощутимый сдвиг** для пользователя или для команды (релизный инкремент). Внутри фазы пункты короткие; крупные спеки — в `docs/features/` и `docs/microservices/`.

---

## Текущее состояние кодовой базы

Репозиторий на этапе **документации и фундамента Фазы 0**: прикладного кода сервисов (Java/Go/Flutter) **нет** — только спецификации, контракты и инфраструктурный каркас.

| Компонент | Состояние |
|-----------|-----------|
| Документация `docs/` | Актуальные спеки, план фаз, микросервисы |
| `protos/` | Protobuf (S2S: `voice/s2s/v1/s2s.proto`); **buf** — [buf.work.yaml](../buf.work.yaml), [protos/buf.yaml](../protos/buf.yaml), CI |
| `src/frontend`, `src/backend`, `src/admin` | Зарезервированная структура монорепо; README в каталогах |
| `src/backend/migrations/` | Первая волна SQL под [DATA_SCOPE_V1.md](DATA_SCOPE_V1.md) (`auth_db`, `user_db`, `social_db`, `chat_db`, `messaging_db`) |
| Docker Compose | PostgreSQL (несколько БД) + Redis для локального стенда ([README.md](../README.md)) |
| CI | GitHub Actions: buf lint/format/breaking (PR), валидация `docker compose`; проверка ссылок в `docs/` — [docs-link-check.yml](../.github/workflows/docs-link-check.yml) при изменениях в доках |

Целевые сервисы и БД по-прежнему описаны в [MICROSERVICES.md](MICROSERVICES.md) и `docs/microservices/*`; ниже — **куда класть код** при появлении реализации.

## Фаза 0 — Фундамент

**Цель:** инфраструктура, без которой нельзя строить фичи.

- [ ] **Схемы БД (первая волна)** — миграции под Фазу 0 + 1: `auth_db`, `user_db`, `social_db`, `chat_db`, `messaging_db` — [DATA_SCOPE_V1.md](DATA_SCOPE_V1.md), детали целевых таблиц — секции «Модель данных» в [microservices/](microservices/)
- [ ] **API Gateway** — REST + WebSocket, маршрутизация
- [ ] **CI/CD** — GitHub Actions: lint, тесты, Docker, деплой на staging
- [ ] **Docker Compose (dev)** — сервисы + PostgreSQL + Redis одной командой
- [ ] **Auth: доработка** — refresh (opaque, 30 дней), logout, валидация, тесты
- [ ] **Общая библиотека Go** — JWT, middleware, логирование, конфиг
- [ ] **Flutter: скелет** — three-column layout, бэкенд вместо Firebase, DI, state
- [ ] **i18n-каркас** — ARB, EN+RU, `flutter gen-l10n` — [features/i18n.md](features/i18n.md)

**Результат:** локальный стенд, логин через бэкенд. Процесс: [CONTRIBUTING.md](CONTRIBUTING.md), [TESTING.md](TESTING.md), [DEPLOYMENT.md](DEPLOYMENT.md).

### Первый вертикальный срез (после фундамента Фазы 0, до полного закрытия Фазы 1)

**Цель:** один сквозной сценарий для проверки стенда, CI и контрактов: **Auth (Java) + API Gateway + один продуктовый поток** (например регистрация/логин и выдача JWT через Gateway, или health + валидация токена — конкретику зафиксировать в PR при старте реализации).

- Выполняется **после** каркаса: Compose, миграции v1 в репозитории, buf/CI, скелеты `src/*`.
- Не заменяет полный чеклист Фазы 1 (DM, друзья, Realtime, Messaging) — это следующий объём работ.

Чеклист выката и стендов: [DEPLOYMENT.md](DEPLOYMENT.md).

## Фаза 1 — MVP: личные сообщения

**Цель:** регистрация, друг, переписка 1-на-1.

### Бэкенд

- [ ] **Messaging** — отправка/получение; **базовое** edit/delete (одна политика, без «для всех/себя» и без метки «(ред.)» в UI — **Фаза 3**); PostgreSQL, курсор, REST
- [ ] **Realtime Service** — JWT, доставка сообщений, Redis Pub/Sub, `s` + `resume`, догрузка через Messaging — [ARCHITECTURE_REQUIREMENTS.md](ARCHITECTURE_REQUIREMENTS.md)
- [ ] **Чаты (DM)** — создание диалогов, список с превью и unread, `mark_read`
- [ ] **Друзья** — запрос, accept/decline, список, блок
- [ ] **Профиль (базовый)** — имя, аватар (R2), «О себе»
- [ ] **Presence** — онлайн/оффлайн по WS, последний визит

### Клиент (Web)

- [ ] Логин / регистрация, список диалогов, чат в реальном времени
- [ ] Поиск пользователей, запрос в друзья, профиль, индикатор онлайн
- [ ] **Локализация** — экраны Фазы 1 на EN+RU

### Не входит

Markdown, реакции, треды, пины, вложения, typing, push, звонки.

**Результат:** веб-мессенджер 1-на-1 для первых тестеров.

---

## Фаза 2 — Звонки 1:1

**Цель:** позвонить другу.

- [ ] **LiveKit** — сервер, выдача токенов
- [ ] **1:1** — исходящий/входящий (WS), accept/decline
- [ ] **Signaling** — offer/answer/ICE через бэкенд (не Firebase)
- [ ] **Клиент** — экран звонка, mute, speaker, hangup

**Результат:** голос поверх уже работающих DM.

---

## Фаза 3 — Медиа и «живой» чат

**Цель:** картинки/файлы в переписке + ощущение «живого» диалога.

- [ ] **R2 + presigned** — лимит 50MB (free)
- [ ] **Вложения** — картинки (thumb + full), файлы (иконка, имя, размер)
- [ ] **Изображения** — WebP, thumbnail
- [ ] **ClamAV** — после появления вложений
- [ ] **Typing indicator** — throttle ~3 с
- [ ] **Статусы доставки** — отправлено / доставлено / прочитано
- [ ] **Edit/delete в UX** — «(ред.)», для всех / для себя (согласовано с API)

**Результат:** базовый паритет с Telegram по медиа и сигналам в DM.

---

## Фаза 4 — Групповые чаты

**Цель:** общение компанией, текст + голос в группе.

- [ ] **Группы** — до ~500 участников, invite/kick, аватар
- [ ] **Роли (простые)** — owner и участники
- [ ] **Групповой голос** — до 32, join к активному звонку
- [ ] **Forward** — с атрибуцией
- [ ] **Реакции** — эмодзи, счётчики
- [ ] **In-app уведомления** — звук, badge (без FCM — он в Фазе 6)

**Результат:** «пати в одном чате».

---

## Фаза 5 — Спейсы: структура

**Цель:** сервер как у Discord — каркас сообщества.

- [ ] **Спейс** — создание, иконка, описание
- [ ] **Дерево** — `space_tree_nodes`: текстовые чаты (Chat) и голосовые комнаты в одном порядке сортировки, категории
- [ ] **Инвайты** — ссылка, срок, лимит использований
- [ ] **Роли** — Owner → Admin → Mod → Member → Guest
- [ ] **Модерация спейса** — кик, бан, мьют, slow mode
- [ ] **Клиент** — sidebar спейсов, дерево текстовых чатов (группы + каналы), голосовые комнаты (join/leave, участники)

**Результат:** можно жить в спейсе; чат в текстовом чате спейса ещё без Markdown/пинов и без push.

---

## Фаза 6 — Спейсы: чат и FCM

**Цель:** нормальный чат в группах/каналах и пуши Web/Android.

- [ ] **Markdown** — жирный, курсив, код, цитаты, ссылки
- [ ] **Упоминания** — @user, @everyone, @here
- [ ] **Пины**
- [ ] **FCM** — Web + Android: токены, новые сообщения и @mentions; база для матчмейкинга (**Фаза 7**)
- [ ] **Группировка push** — по чату, где позволяет платформа

**Результат:** спейс ощущается «готовым» на вебе/Android; iOS-push — **Фаза 8**.

---

## Фаза 7 — Матчмейкинг

**Цель:** поиск тиммейтов (ключевое УТП).

- [ ] **Каталог игр** — роли/ранги в конфиге
- [ ] **Профиль игрока** — игры, ранг, роли, регион
- [ ] **Очередь** — критерии и слоты
- [ ] **Подбор** — матч-отряд (текст + голос); уведомление через **FCM (Фаза 6)**; iOS — после **APNs (Фаза 8)**, до тех пор in-app/WS
- [ ] **Таймаут** — например 30 мин без подтверждения
- [ ] **Рейтинг** — звёзды после матча
- [ ] **История матчей**

**Результат:** уникальная фича для геймеров.

---

## Фаза 8 — Мобильные клиенты и push

**Цель:** не только веб; iOS и доводка уведомлений.

- [ ] **APNs** — обычные пуши (тот же контур, что FCM)
- [ ] **VoIP** — PushKit + CallKit для входящих на iOS
- [ ] **FCM** — доработка под мобилу, согласованность с APNs
- [ ] **Flutter mobile** — layout, Android / iOS релизы
- [ ] **Оффлайн-кэш** — последние N сообщений на чат (SQLite)
- [ ] **Windows** — desktop-сборка, auto-updater

**Результат:** мультиплатформа для основной аудитории.

---

## Фаза 9 — Поиск

**Цель:** найти сообщение, человека, публичный спейс.

- [ ] **Полнотекст** — PostgreSQL tsvector/GIN v1, в чате и глобально
- [ ] **Пользователи и спейсы** — каталог публичных спейсов

**Результат:** продукт переживаем при росте истории и серверов.

---

## Фаза 10 — Глубина сообщества

**Цель:** тяжёлые сценарии текстовых чатов и голоса.

- [ ] **Треды** — ветки; дефолт у канала — тред-ориентированно, у группы — по настройкам (всё переопределяется)
- [ ] **Шаринг экрана** — до 3 стримов в войсе
- [ ] **Shared media** — вкладки медиа/файлы/ссылки в инфо чата
- [ ] **Кастомные роли** — гранулярные права (32+), overrides на текстовый чат (`chat_id`) и голос

**Результат:** крупные активные сообщества комфортны.

---

## Фаза 11 — Доверие (базовая)

**Цель:** минимум безопасности **до** приёма платежей (**Фаза 12**).

- [ ] **Репорты (базовые)** — UI + API, категории, `moderation_db`, очередь. Спека: [features/reports.md](features/reports.md). **Без** модераторской панели — **Фаза 14**
- [ ] **2FA** — TOTP
- [ ] **Приватность профиля** — пресеты + кастом — [features/privacy.md](features/privacy.md)

**Результат:** можно открывать монетизацию без стыда за отсутствие жалоб и 2FA.

---

## Фаза 12 — Монетизация: биллинг

**Цель:** деньги за подписки.

- [ ] **Premium** — статус, аним. аватар, баннер, лимиты файлов 200MB, 5 профилей
- [ ] **Space Pro** — лимиты спейса, кастомные эмодзи
- [ ] **Платежи** — Paddle (не-СНГ), CloudPayments (РФ)
- [ ] **Годовая подписка** — скидка 20%

*Перед прод-платежами — закрыть Фазу 11.*

**Результат:** revenue stream, freemium без урезания базы.

---

## Фаза 13 — Профили и верификация

**Цель:** мульти-профиль и доверие к личностям/брендам.

- [ ] **Мульти-профиль** — 2 бесплатно / 5 в Premium, раздельные контакты и спейсы
- [ ] **Верификация** — [features/verification.md](features/verification.md): личная (Twitch Partner / YouTube YPP), организации (DNS TXT + fallback), значки в UI, приоритет в поиске

**Результат:** платный слой и репутация связаны с аккаунтом, не только с косметикой подписки.

---

## Фаза 14 — Модерация платформы

**Цель:** масштаб глобальной модерации.

Граница с **Фазой 11**: там подача и очередь; здесь — операции и авто-логика — [features/reports.md](features/reports.md).

- [ ] **Авто-мод** — пороги, shadow-ban, типы очередей
- [ ] **Панель модераторов** — отдельный веб: разбор, санкции, аудит, раздельные потоки (в т.ч. спейсы)

**Результат:** модерация выдерживает рост.

## Фаза 15 — E2E в DM (опционально)

**Цель:** усиленная приватность для тех, кому нужно.

- [ ] **Signal Protocol** — опционально для DM, ограничения поиска — [features/encryption.md](features/encryption.md)

**Результат:** E2E как opt-in, не ломая продукт для остальных.

## Фаза 16 — Боты

**Цель:** расширяемость для разработчиков.

## Фаза 17 — Сторис

**Цель:** короткий контент и связь с ММ.

Спека: [features/stories.md](features/stories.md); API: [microservices/story-service.md](microservices/story-service.md).

- [ ] **Контент + редактор** — фото/видео/текст/клип, оверлеи, стикеры, дудл, фильтры, трим, R2/File
- [ ] **Лента** — 24ч, архив 30 дней, просмотры, реакции, реплаи, кольцо на аватаре
- [ ] **Highlights** — подборки на профиле, своя приватность
- [ ] **Матчмейкинг** — «Ищу пати», пост-матч стори — [features/matchmaking.md](features/matchmaking.md)

**Результат:** контентный слой для удержания и LFG.

## Фаза 18 — Рост и доступность

**Цель:** проще зайти и пользоваться.

- [ ] **Deep links** — `voice://`, `https://voice.gg/…`
- [ ] **Онбординг** — 4–5 шагов, контекстные подсказки
- [ ] **A11y** — клавиатура, screen reader, контраст, reduced motion

**Результат:** меньше трения для новичков и пользователей ВАТ.

- [ ] **Синхронизация** — роли, баны, события
- [ ] **Дефедерация**
