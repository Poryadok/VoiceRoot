# Verification — верификация аккаунтов

## Два типа верификации

| Тип         | Кому                                         | Значок                | Как получить                |
|-------------|----------------------------------------------|-----------------------|-----------------------------|
| Личная      | Стримеры, про-игроки, публичные личности     | ✅                     | Через привязанные платформы |
| Организация | Студии, издатели, киберспорт-команды, бренды | 🏢 (иконка, не символ) | DNS-верификация домена      |

Конкретный дизайн значков — отдельное решение (UI). Требование: значки — **системные иконки**, не Unicode-символы, чтобы их нельзя было сымитировать в нике.

---

## Личная верификация — через платформы

Автоматически. Пользователь привязывает аккаунт платформы → система проверяет статус → выдаёт значок.

### Поддерживаемые платформы (V1)

| Платформа | Условие                                                                |
|-----------|------------------------------------------------------------------------|
| Twitch    | Partner (не Affiliate — это массовый статус, значок потеряет ценность) |
| YouTube   | Участник YouTube Partner Program с верифицированным каналом            |

### Поддерживаемые платформы (будущее)

- FACEIT (верификация ранга для CS2 и других)
- ESL / FACEIT Pro League — про-статус
- Riot Games (API для верифицированных про-игроков)
- Twitter/X — если статус верификации стабилизируется

### Логика

1. Пользователь идёт в Настройки → Связанные аккаунты → подключает Twitch
2. OAuth → Voice получает токен → проверяет `broadcaster_type == "partner"`
3. Если да — значок выставляется и привязывается к платформе
4. Если пользователь отвязывает аккаунт или теряет Partner-статус → значок снимается автоматически (проверка по крону)

## Верификация организаций — DNS

Автоматизированная, но с ручным fallback для крупных партнёров.

### Процесс (DNS)

1. Организация создаёт аккаунт Voice с типом "Организация"
2. Указывает официальный домен (например: `riotgames.com`)
3. Система генерирует уникальную TXT-запись: `voice-verify=abc123xyz`
4. Владелец добавляет её в DNS
5. Система проверяет DNS → подтверждает → выдаёт значок

DNS-верификация доказывает владение доменом без ручной проверки с нашей стороны. Используется в Google Search Console, GitHub, Twitch для директорий.

### Ручной процесс (fallback)

Для организаций без домена или крупных партнёров, которые хотят приоритетную поддержку:
- Заявка на email → ручная проверка → approval

## Анти-спуфинг

Верификация бессмысленна если можно назвать себя "Fаker" с кириллической "а".

### Меры

1. **Значки — только системные иконки** в UI (SVG/иконка), никаких Unicode-символов которые пользователь мог бы скопировать в ник
2. **Нормализация никнеймов**: при регистрации ник проходит Unicode normalization (NFKC) + замена гомоглифов на латиницу для проверки уникальности
3. **Поиск по нормализованному имени**: поиск "Faker" находит все варианты гомоглифов → сверху — верифицированный аккаунт

## Что даёт верификация

| Фича                                                                   | Есть                         |
|------------------------------------------------------------------------|------------------------------|
| Значок рядом с именем во всех чатах                                    | ✓                            |
| Приоритет в результатах поиска по никнейму                             | ✓                            |
| Приоритет в каталоге спейсов (если у спейса верифицированный владелец) | ✓                            |
| Особые права или расширенные лимиты                                    | ✗ — только косметика и поиск |

## Что НЕ делаем

- **Платная верификация** — репутационные риски, обесценивает значок
- **Верификация через email-домен** — слабее DNS (любой может создать почту на домене)
- **Верификация Twitter/X** — нестабильный статус с 2022 года

## Service Ownership

- Owner: `User Service` (verification status/badge state в профиле)
- Интеграции identity: `Auth Service` (OAuth-linked identities, токены и проверки)

## Enforcement path

1. Пользователь инициирует привязку платформы/домена через API.
2. `Auth Service` выполняет OAuth identity checks и подтверждает внешнюю идентичность.
3. `User Service` фиксирует verification state и управляет отображением badge.
4. Плановый re-check статуса (cron) снимает badge при потере условий верификации.
