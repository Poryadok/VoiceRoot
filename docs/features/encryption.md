# Encryption — шифрование

## Решение

Два уровня шифрования с разной зоной ответственности:

| Уровень         | Что защищает                           | Статус                     |
|-----------------|----------------------------------------|----------------------------|
| Transport (TLS) | Трафик клиент → сервер                 | Обязательно везде          |
| E2E             | Содержимое сообщений (сервер не видит) | Опционально, только для DM |

---

## Transport encryption (TLS)

Весь трафик зашифрован на транспортном уровне:
- HTTPS для REST API
- WSS для WebSocket
- TLS для gRPC

Сервер при этом видит plaintext — это необходимо для серверного поиска, уведомлений, модерации. Защищает от перехвата на уровне сети (Wi-Fi роутер, провайдер, MITM).

---

## E2E шифрование

E2E — опциональная функция, **только для DM** (личная переписка между двумя пользователями). Для групп, каналов и спейсов не предусмотрено.

### Включение

Пользователь явно включает E2E для конкретного DM-чата. При включении система показывает предупреждение:

> **Включён режим сквозного шифрования**
> Сообщения в этом чате зашифрованы и недоступны серверу.
> — Глобальный поиск по сообщениям этого чата не работает.
> — Локальный поиск работает только среди загруженной истории на этом устройстве.

### Ограничения поиска в E2E-чатах

| Тип поиска                     | Работает?                                          |
|--------------------------------|----------------------------------------------------|
| Глобальный поиск по сообщениям | Нет — сервер не видит plaintext                    |
| Поиск внутри чата (серверный)  | Нет                                                |
| Поиск внутри чата (локальный)  | Да, но только по загруженной истории на устройстве |

Поиск по имени контакта в глобальном поиске продолжает работать — шифруется только содержимое сообщений.

### Offline cache на устройстве (at-rest)

Решение: **локальная SQLite offline cache шифруется целиком** (SQLite3MultipleCiphers / SQLCipher-совместимый режим через `PRAGMA key`).

| Аспект | Политика |
|--------|----------|
| Что кэшируется | Расшифрованные тела сообщений (в т.ч. E2E после decrypt) — для офлайн-чтения и локального поиска |
| At-rest | Файл БД на диске зашифрован; ключ генерируется при первом открытии и хранится в OS secure storage (`flutter_secure_storage`) |
| Web | Персистентный offline cache не используется (in-memory, как в Phase 8) |
| Миграция | Существующий незашифрованный `voice_message_cache.sqlite` при первом запуске с шифрованием конвертируется (rekey), затем старый файл удаляется |
| Logout | Кэш очищается по `profileId` / `clearAll` как сейчас; ключ БД на устройстве сохраняется до удаления приложения |

Сервер по-прежнему не видит plaintext E2E; риск чтения кэша с диска без ключа ОС/устройства снижается по сравнению с открытым SQLite.

### Доверие к ключам (TOFU и верификация)

Решение для v1 (Фаза 15):

| Аспект | Политика |
|--------|----------|
| **TOFU** | По умолчанию — неявный (Trust On First Use): при первом E2E-контакте клиент доверяет identity key собеседника из pre-key bundle на сервере. Отдельная верификация при включении E2E **не** обязательна. |
| **Код верификации** | Опционально, в **инфо DM** (настройки чата). Не блокирует включение E2E. Это **короткий отпечаток**, не полный 60-значный safety number Signal. |
| **Смена identity key** | Баннер в чате, когда у собеседника новый ключ (переустановка, новое устройство без бэкапа и т.д.). Действия: принять новый ключ или не доверять (новые сообщения не расшифровываются до решения). |

#### Формат кода верификации

- **6 символов**, группы по 2: `XX-XX-XX` (например `K7-M2-9P`).
- Алфавит: Crockford Base32 (`A–Z`, `2–9`; без `0`/`O`, `1`/`I`/`L`) — меньше путаницы при сверке голосом.
- **Один и тот же код у обоих** участников DM: симметрично из пары identity keys и стабильных id (`profile_id`), затем усечение displayable fingerprint libsignal (или эквивалентный hash) до 6 символов.
- Не совместим с QR/кодом Signal или WhatsApp — отдельный display format Voice.

Энтропия короче полного safety number (~24–30 bit); достаточно для opt-in DM и сверки «друг переустановился?», но слабее против целенаправленного MITM на сервере. В UI не позиционировать как «военный» уровень Signal.

### Что ещё отсутствует в E2E-чатах

- Серверная модерация содержимого
- Работа с нескольких устройств одновременно (Signal Protocol поддерживает multi-device — в v1 не реализуется)

---

## Протокол E2E

Используется **Signal Protocol (libsignal)**:
- Double Ratchet Algorithm — обеспечивает forward secrecy (PFS): компрометация одного ключа не раскрывает прошлые сообщения
- X3DH (Extended Triple Diffie-Hellman) — начальный обмен ключами
- Есть Dart/Flutter binding для мобильного клиента
- Стандарт индустрии: WhatsApp, Signal, Telegram Secret Chats

## Резервное копирование ключей

Encrypted key backup — ключи шифруются пользовательским паролем и хранятся на сервере:
- Сервер видит только зашифрованный blob, не имея доступа к ключу шифрования
- Ключ шифрования — у пользователя (на основе его пароля/PIN)
- При переустановке приложения пользователь вводит пароль → ключи восстанавливаются → история E2E-чатов доступна
- Аналог: Telegram Cloud Password для Secret Chats backup

## Файлы в E2E-чатах

- Файлы хранятся на сервере в зашифрованном виде (сервер видит только blob)
- Ретеншен: 90 дней для E2E-чатов независимо от тарифа (синхронизировано с `docs/microservices/file-service.md`)
- По истечении срока зашифрованный blob удаляется автоматически
- Пользователи в E2E-чате информируются об автоудалении файлов через 90 дней

### E2E на новом устройстве без бэкапа

Если пользователь устанавливает приложение на новое устройство и **не настроил key backup** или **пароль бэкапа утерян**:
- История E2E-чатов до момента подключения нового устройства **недоступна** — ключи расшифровки не существуют на новом устройстве
- В чате вместо сообщений показывается плейсхолдер:
  > *"Сообщения до [дата] зашифрованы и недоступны на этом устройстве. Настройте резервную копию ключей, чтобы не потерять историю."*
- Новые сообщения после подключения устройства расшифровываются нормально (Double Ratchet создаёт новые ключи)

## Осознанное решение

E2E не делается "везде" намеренно:
- Серверный поиск, модерация, уведомления требуют доступа к plaintext
- Для защиты трафика TLS достаточен в большинстве сценариев
- E2E предлагается как opt-in для пользователей, которым важна приватность DM выше удобства поиска

## Service Ownership

- Owner: `Messaging Service` (E2E message flow) + `File Service` (E2E attachment lifecycle/retention)
- Key backup orchestration: `Auth Service` (учётный контур и безопасное хранение encrypted key blob)

## Enforcement path

1. `API Gateway` применяет transport-level TLS и проксирует запросы.
2. `Messaging Service` применяет ограничения E2E (только DM, без server-side search/moderation).
3. `File Service` применяет E2E retention policy (90 дней всегда) и авто-expiry.
4. `Search Service` исключает E2E content из серверного индекса.
