# Updates — обновление клиентов и версионирование

## Проблема

Бэкенд и клиенты развиваются независимо. Нужно:
- знать, какие версии клиентов совместимы с текущим API;
- принудительно блокировать устаревшие клиенты;
- доставлять обновления без визита в стор (десктоп, OTA-патчи).

---

## Версии и совместимость

### Что хранит бэкенд

На бэке есть конфигурация (хранится в БД или конфиге деплоя, меняется без редеплоя) per-платформа:

| Поле                    | Пример                                               | Смысл                                                 |
|-------------------------|------------------------------------------------------|-------------------------------------------------------|
| `platform`              | `android`, `ios`, `windows`, `macos`, `linux`, `web` | Идентификатор платформы                               |
| `min_supported_version` | `1.4.0`                                              | Ниже этого — принудительный апдейт, API отказывает    |
| `latest_version`        | `1.7.2`                                              | Ниже этого — мягкое предупреждение, апдейт опционален |
| `update_url`            | CDN-ссылка или store deep-link                       | Куда вести пользователя                               |
| `release_notes`         | строка                                               | Краткий changelog для показа в промпте                |
| `shorebird_patch`       | `42`                                                 | Номер последнего Shorebird-патча (mobile only)        |

### Эндпоинт проверки версии

```
GET /api/v1/version?platform={platform}&version={semver}&shorebird_patch={n}
```

Ответ:
```json
{
  "force_update": false,
  "update_available": true,
  "latest_version": "1.7.2",
  "min_supported_version": "1.4.0",
  "update_url": "https://...",
  "release_notes": "Исправлен войс на Android 14, новые эмодзи-реакции",
  "shorebird_patch": 42
}
```

Если `force_update: true` — API продолжает работать только для этого `/version` эндпоинта.
Все остальные запросы возвращают `426 Upgrade Required` с body:
```json
{ "error": "client_outdated", "update_url": "https://..." }
```

## Клиентская логика

### При старте приложения

1. Клиент запрашивает `/api/v1/version` со своей версией и платформой.
2. **`force_update: true`** — показывает fullscreen-блокировку с кнопкой "Обновить", приложение не работает.
3. **`update_available: true`** — показывает промпт (не fullscreen), пользователь может отложить.
4. **Оба false** — silent, никаких UI-элементов.

Повторная проверка — раз в 4 часа в фоне (не блокирует UI).

### Состояния UI

```
┌─────────────────────────────────────────┐
│  Обязательное обновление                │   ← force_update
│  Версия 1.3.x больше не поддерживается  │
│  [Обновить сейчас]                      │
└─────────────────────────────────────────┘

┌─────────────────────────────────────────┐
│  Доступно обновление 1.7.2              │   ← update_available
│  "Исправлен войс на Android 14"         │
│  [Обновить]  [Позже]                    │
└─────────────────────────────────────────┘
```

## Платформа: Web

- Всегда актуальная версия через CDN + Service Worker cache invalidation.
- Версионирование API (v1, v2) — единственный механизм совместимости.
- Проверка `/api/v1/version` не нужна — браузер всегда грузит свежий код.

---

## Платформа: Desktop (Windows, macOS, Linux)

**Механизм:** `auto_updater` (Flutter, Sparkle/WinSparkle под капотом) или кастомный.

**Флоу:**
1. При старте и раз в 4 часа — проверка `/api/v1/version`.
2. Если `update_available: true` — скачиваем новый инсталлятор/пакет в фоне без уведомления.
3. После завершения скачивания — промпт в стиле Discord:
   ```
   Доступно обновление 1.7.2
   "Исправлен войс на Android 14, новые эмодзи-реакции"
   [Перезапустить и обновить]  [Позже]
   ```
4. "Позже" — промпт исчезает, обновление применится при следующем перезапуске.
5. "Перезапустить и обновить" — приложение перезапускается, применяет патч, стартует новая версия.

**Дистрибуция:**
- Файлы обновлений хранятся на CDN (S3 / Cloudflare R2).
- Подпись обновлений обязательна (Sparkle EdDSA / MSIX-сертификат).
- `update_url` в ответе `/version` указывает на manifest (не на бинарь напрямую), auto_updater сам разбирает delta-пакет.

## Платформа: Mobile

### Android

| Механизм                    | Когда                                                                     |
|-----------------------------|---------------------------------------------------------------------------|
| Google Play Store           | Основной канал, крупные релизы                                            |
| Play Core In-App Update API | Показывает нативный промпт обновления внутри приложения без выхода в стор |
| Shorebird OTA               | Только Dart-код, мелкие фиксы, без стор-ревью                             |

**Флоу Play Core:**
- `FLEXIBLE` (гибкое) — скачивается в фоне, промпт "применить" после скачивания.
- `IMMEDIATE` (немедленное) — fullscreen overlay, обязательное; используется при `force_update: true`.

**Флоу Shorebird:**
- На старте Shorebird Runtime проверяет патч-сервер.
- Патч (`shorebird_patch` из `/version`) скачивается в фоне.
- Применяется при следующем запуске — пользователь не замечает.
- Не заменяет стор-релиз для нативных изменений (permissions, plugins).

### iOS

| Механизм      | Когда                                                   |
|---------------|---------------------------------------------------------|
| App Store     | Основной канал                                          |
| In-App Update | Редирект на App Store (нативного аналога Play Core нет) |
| Shorebird OTA | Только Dart-код, то же что Android                      |

**Флоу:**
- `force_update: true` → открывает App Store через `StoreKit` overlay или `itms-apps://` deep link.
- `update_available: true` → промпт с кнопкой "Обновить" → открывает App Store.

## Staged rollout — пост-v1

Постепенная раскатка (5% → 25% → 100%) реализуется после v1:
- Поле `rollout_percentage` в конфиге версии.
- Клиент включается в когорту по хэшу `user_id`.
- До достижения порога пользователи из неактивной когорты видят `update_available: false`.

## Backend: сервис и хранение

- Конфигурация версий хранится в отдельной таблице `client_versions` в основной БД.
- CRUD-эндпоинты закрыты admin-ролью.
- Кэш ответа `/version` — 5 минут (Redis), не нагружает БД.
- Изменение `min_supported_version` или `latest_version` — инвалидирует кэш немедленно.

```sql
CREATE TABLE client_versions (
  platform          VARCHAR(20) PRIMARY KEY,
  min_supported     VARCHAR(20) NOT NULL,
  latest_version    VARCHAR(20) NOT NULL,
  update_url        TEXT        NOT NULL,
  release_notes     TEXT,
  shorebird_patch   INT,
  updated_at        TIMESTAMPTZ DEFAULT NOW()
);
```

## Метрики

- `client_version_distribution` — гистограмма версий клиентов (по платформам).
- `force_update_blocks` — сколько раз отдали 426.
- `update_prompt_accepted` / `update_prompt_deferred` — конверсия промпта.
- `shorebird_patch_adoption` — доля устройств на последнем патче.
