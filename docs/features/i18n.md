# Локализация (i18n) — интернационализация

## Языки

| Версия | Языки                      |
|--------|----------------------------|
| v1     | English (en), Русский (ru) |
| Позже  | По мере спроса — любые     |

Новый язык добавляется без изменений кода: только новый ARB-файл с переводами.

## Клиент (Flutter)

**Стек:** `flutter_localizations` + `intl` + ARB-файлы.

- Строки хранятся в `lib/l10n/app_en.arb`, `app_ru.arb` и т.д.
- Плюрализация через ICU MessageFormat — покрывает сложные правила русского языка (1 сообщение / 2 сообщения / 5 сообщений)
- Форматирование дат и чисел через `intl` — locale-aware автоматически
- Язык: берётся из системных настроек; ручная смена в настройках приложения
- Язык хранится в профиле пользователя и синхронизируется между устройствами

```
lib/
  l10n/
    app_en.arb
    app_ru.arb
    app_de.arb   ← пример будущего добавления
```

## Бэкенд

- Системные сообщения (уведомления, email) — генерируются на языке пользователя; язык берётся из профиля
- API возвращает данные as-is, без перевода (контент пользователей не переводится сервером)
- Сообщения об ошибках от API — на английском (локализуются на клиенте по error code)

## Full-text Search

PostgreSQL `tsvector` требует языковой конфигурации. Проблема в том, что в одном чате могут быть сообщения на разных языках.

**Решение для v1:**
- Использовать `pg_trgm` (trigram-индекс) — language-agnostic, работает для EN и RU без специальной конфигурации
- Поиск через `ILIKE '%query%'` с trigram-ускорением (`GIN` индекс)
- Качество поиска немного хуже чем `tsvector` (нет стемминга), но покрывает все языки одинаково

**При масштабировании:**
- Если нужен стемминг — определять язык сообщения на этапе индексации (langdetect) и хранить в отдельной колонке `lang`; использовать соответствующий `tsvector` конфиг
- Или перейти на Elasticsearch/Meilisearch с multi-language поддержкой из коробки

## RTL (Right-to-Left)

Арабский, иврит и другие RTL-языки требуют отдельного архитектурного решения во Flutter:
- Все отступы через `EdgeInsetsDirectional` вместо `EdgeInsets`
- Layout через `start`/`end` вместо `left`/`right`
- Ряд иконок/элементов нужно зеркалить

**Решение: RTL не поддерживается в v1.** При добавлении RTL-языка — отдельный технический тред. Закладывать с самого начала не нужно, но не использовать хардкод `left`/`right` там, где это легко избежать.

## Что НЕ входит в scope

- Machine translation сообщений (как в Telegram Premium) — отложено
- Локализация пользовательского контента (названия спейсов, каналов) — остаётся как есть
- Региональные версии приложения с разным функционалом — нет
